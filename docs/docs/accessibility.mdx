# 可访问性

LDS系统在设计时遵循的一个原则是：基于LDS开发体系构建的所有应急行业类的应用，可以为随时随地、方便快捷的为各类人群提供信息服务。
WEB应用开发中的可访问性意味着让每个人都可以访问相同的信息，无论个人的身体和认知能力以及访问WEB的方式是否存在一定的限制。

### 什么是A11Y

A11Y是英文单词可访问性“Accessibility”的缩写。针对Web应用的可访问性W3C提供了WCAG(2)标准(Web Content Accessibility Guidelines)。
符合该标准的Web应用可以最大限度的满足残疾人获取Web应用中各类信息的需求。

### LDS系统中的可访问性

LDS开发体系中的组件在开发过程中，会遵循以下可访问性功能点检查清单。
LDS核心开发人员在开发元组件库、业务组件库的时候，各组件需要通过检查清单中的各项检测后才可以正式发布。

### 通用规则

- [ ] 组件需要通过Chrome浏览器的Axe插件的自动检测
- [ ] 组件需要通过Chrome浏览器的Wave插件的自动检测
- [ ] 在Windows环境下，可以兼容NVDA屏幕阅读软件
- [ ] 在Windows环境下，可以兼容Jaws屏幕阅读软件
- [ ] 在MacOS环境下，可以兼容VoiceOver屏幕阅读软件
- [ ] 在iOS环境下，可以兼容VoiceOver屏幕阅读软件
- [ ] 在Android环境下，可以兼容TalkBack屏幕阅读软件
- [ ] 当应用中的文本大小增加到200%时，应用可以正常工作
- [ ] 当应用界面放大400%时，应用可以正常工作

### 内容

- [ ] 应用中的文本描述使用通俗易懂的语音，避免使用比喻、习语和复杂的隐喻
- [ ] 按钮，链接和标签元素的内容是唯一的和描述性的
- [ ] 每个页面只有一个&lth1&gt标题
- [ ] 标题元素的出现顺序具备逻辑性
- [ ] 标题的等级不能跳级
- [ ] 列表内容使用列表元素（&ltol&gt, &ltul&gt, &ltdl&gt）来实现
- [ ] 主要内容的行高最少是字体大小的1.5倍

### 标签

- [ ] &lthtml&gt元素需要有正确的lang特性
- [ ] 每个页面都需要有唯一的&lttitle&gt标签
- [ ] 使用标记来区分页面的不同重要区域，比如：使用&ltnav&gt元素来标记站点的导航区域
- [ ] 确保内容可以线性访问。元素tabindex特性值应该只是0或-1
- [ ] 标签避免使用autofocus特性
- [ ] 避免使用&lttooltips&gt标签的tltle特性
- [ ] 所有的链接都使用&lta&gt标签
- [ ] 所有的按钮都使用&ltbutton&gt元素
- [ ] 使用基于&ltsvg&gt的icons，避免使用字体icons
- [ ] 表格形式的数据，使用&lttable&gt元素实现
- [ ] 表格头使用&ltth&gt元素
- [ ] 表格标题使用&ltcaption&gt元素

### 键盘

- [ ] 所有的可交互元素都具有可视化的焦点样式
- [ ] 键盘焦点顺序和可视化布局相匹配
- [ ] 没有不可见的焦点元素
- [ ] 按钮可以通过空格或者回车键访问
- [ ] 菜单或者弹窗可以通过esc键关闭

### 图像

- [ ] 所有的&ltimg&gt元素都需要alt特性值
- [ ] 复杂的可视化图形需要有可代替的文本
- [ ] 装饰性的&ltsvg&gt元素使用aria-hidden=“true”
- [ ] &ltimg&gt元素的格式为svg时，使用role="img"属性

### 表单

- [ ] 所有的&ltinput&gt元素都需要有一个对应的&ltlabel&gt元素
- [ ] 表单内容划分为多个部分时，使用&ltfieldset&gt分组，并使用&ltlegend&gt为分组提供说明
- [ ] 表单的输入框提供自动完成功能
- [ ] 输入框的错误提示和输入框相关联

### 颜色和对比度

- [ ] 所有文本内容颜色对比度至少为4.5:1
- [ ] 所有界面元素的对比度至少为3:1
- [ ] 界面通过色盲仿真器的测试
- [ ] 检查覆盖在图片或者视频上的文本是否清晰

### 移动设备支持

- [ ] web站点可以在移动设备上适应各种朝向
- [ ] web站点可以在最少320px的视图中正常显示
- [ ] 移除水平的滚动功能
- [ ] 可触摸区域最小是44*44px
- [ ] 视图的放大缩小功能没有被禁用
